# YOSAKOI PHOTO ARCHIVE 実装フェーズ計画

## 1. 前提

本ドキュメントは `requirements.md` を要件定義書として読み込み、設計レビュー可能な粒度で実装フェーズ、画面一覧、実装順序、技術的リスクを整理したものです。

この段階ではコード生成を行わず、後続実装の判断材料とします。

## 2. システム全体方針

### 2.1 プロダクト方針

- スマートフォン、特に iPhone 利用を最優先とする。
- Google Photos 風の写真閲覧体験を目指す。
- 会員制の非公開フォトアーカイブとして、安全な閲覧、検索、ダウンロードを提供する。
- 顔認識は行わず、管理者による手動タグ付けで「自分が写っている写真」を探せるようにする。
- ZIP 一括ダウンロードは実装しない。

### 2.2 技術方針

- Backend: Python / FastAPI
- Frontend: Jinja2 / Bootstrap 5 / Vanilla JavaScript
- Database: PostgreSQL
- Object Storage: Cloudflare R2
- Deployment: Docker / Docker Compose / Render
- App: PWA
- Auth: JWT Cookie 認証、bcrypt パスワードハッシュ、30分アイドルタイムアウト
- Image Access: R2 非公開運用、Presigned URL 15分有効

目的:

MVP 実装に入るための土台を整備する。

実装対象:

- FastAPI アプリケーション基本構成
- Jinja2 テンプレート基本構成

- Docker Compose 起動構成

- ログ出力方針

完了条件:

- ローカルで Web アプリと DB が起動できる。
- DB マイグレーションを実行できる。
- トップ画面または疎通確認画面を表示できる。

### Phase 1: 認証・ユーザー管理

目的:

会員制アプリとしてのアクセス制御を成立させる。

実装対象:

- access_log テーブルの最小実装
- dept_user テーブル
- bcrypt によるパスワード保存
- JWT Cookie ログイン
- ログアウト
- ログイン成功・失敗、ログアウト、ダウンロード等のアクセスログ
- 利用期間チェック
- アクセス時の利用期間再確認
- 30分無操作時のCookie/JWT期限切れ
- 管理者によるユーザー作成・編集・有効化・無効化
- CSV 一括登録
- 管理者権限チェック

完了条件:

- 一般ユーザーと管理者のログインができる。
- 期限外ユーザーはログイン不可。
- ログイン中に期限切れとなった場合、強制ログアウトされる。
- 管理者のみユーザー管理画面へアクセスできる。

目的:

写真を年度・イベント単位で整理できるようにする。

目的:

実装対象:

- photos テーブル

- サムネイル自動生成
- Cloudflare R2 保存
- R2 オブジェクトキー設計

- 写真論理削除

完了条件:

- 管理者がアルバムへ複数写真をアップロードできる。
- 原本画像とサムネイル画像が保存される。ローカル開発ではR2オブジェクトキー互換のパスで `LOCAL_STORAGE_ROOT` 配下へ保存する。
- 撮影日時が適切に保存される。
- 削除操作は論理削除となる。

## 6. 技術的リスク

### 6.1 認証・認可

リスク:

- JWT の保存場所によって XSS / CSRF の対策が変わる。
- Jinja2 ベースの画面と API が混在するため、画面遷移と非同期通信の認証方式を揃える必要がある。
- 利用期間切れをログイン時だけでなく各画面/APIアクセス時にも必ず確認する必要がある。

対策:

- HttpOnly Cookie で JWT を保持する方針を優先検討する。
- CSRF トークンをフォーム送信と状態変更 API に適用する。
- 認証依存関数で `dept_user.is_active` と `start_date` / `end_date` を共通チェックする。

### 6.2 Cloudflare R2 と Presigned URL

リスク:

- R2 を公開バケットにすると要件違反となる。
- Presigned URL の発行頻度が多いとレスポンスやコストに影響する。
- URL が HTML に埋め込まれるため、有効期限内の共有を完全には防げない。

対策:

- R2 は非公開バケット前提で設計する。
- サムネイルと原本で URL 発行タイミングを分ける。
- Presigned URL は 15分有効に限定する。
- 画像表示時、ダウンロード時に都度発行する。
- R2 オブジェクトキーは日本語イベント名を直接使わず、年度、アルバムID、写真UUIDを組み合わせる。

### 6.3 画像処理

リスク:

- 大量・高解像度画像のアップロードでメモリ使用量が増える。
- Exif の撮影日時は端末や編集ソフトにより欠落・不正な場合がある。
- 日本語イベント名を R2 パスに含めるとエンコードや運用時の扱いが複雑になる。

対策:

- 画像処理はストリーミングまたは一時ファイル利用を検討する。
- Exif 取得失敗時の補完順序を明確に実装する。
- R2 オブジェクトキーは表示名ではなく、年度、アルバムID、写真UUIDを使って生成する。

### 6.4 スマートフォン UX

リスク:

- 写真グリッド、ズーム、スワイプの挙動が iPhone Safari で崩れる可能性がある。
- PWA のスプラッシュやアイコン要件は iOS と Android で差がある。
- ダウンロード挙動はモバイルブラウザごとに差が出る。

対策:

- 早い段階で iPhone Safari 相当の表示確認を行う。
- PhotoSwipe / Swiper.js の導入可否を Phase 4 で検証する。
- ダウンロードは単体ファイルのみとし、ZIP は実装しない方針を守る。
- Service Worker は認証付きHTMLやPresigned URL付き画像をキャッシュ対象外とし、静的アセット中心に限定する。

### 6.5 DB・検索性能

リスク:

- 写真数が増えると一覧、検索、タグ絞り込みが重くなる。
- お気に入り、ダウンロード済み表示はユーザー別 JOIN が必要になる。

対策:

- 要件のインデックスに加えて、公開・削除条件と検索条件に合わせた複合インデックスを検討する。
- 写真一覧 API はページネーションまたは無限スクロール前提にする。
- 一覧取得では必要なカラムのみ返す。

### 6.6 管理者運用

リスク:

- CSV 一括登録時の重複ログインID、日付不正、文字コード違いで登録エラーが発生しやすい。
- 写真削除が論理削除のため、R2 上の不要ファイルが蓄積する。

対策:

- CSV インポートは事前検証とエラー行表示を行う。
- login_id は一意制約を設定する。
- 物理削除は MVP 後の保守タスクとして別途検討する。

### 6.7 フェーズレビュー

リスク:

- MVP 範囲が広いため、後半フェーズで手戻りが大きくなる可能性がある。

対策:

- Phase 1、Phase 3、Phase 5、Phase 7 の完了時にレビューゲートを設ける。
- 各レビューでは、要件充足、セキュリティ、スマートフォン表示、運用手順を確認する。
